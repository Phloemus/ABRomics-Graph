Adding geosparql to the virtuoso graph is very hard. 
The version that should be used according to the documentation is the versioin 7.2.6 or above.
Meanwhile the version of the graph server image used in docker is virtuoso version 7.2.2.
Using the latest official image of virtuoso (version 7.2.12) doesn't allow to use geosparql
functions (tested: contains, sf_contains, bif:sf_contains, bif:contains)

Multiple plugins seem to be needed to allow geosparql functionalities: 
(described here: https://github.com/openlink/virtuoso-opensource/blob/develop/7/README.GeoSPARQL.md)
The procedure described in the article is on the version 7.2.6 of the virtuoso server. 

The version of the graph installed in the glicid VM is version 7.2.11. 
(github build commit tag: d89671f)
This version of virtuoso should be able to handle geoSPARQL queries with the right config and 
the right plugins.

Performing queries with the isql-v script (which give a SQL promp when executed)
    - Trying to use the bif:sf_contains()       Fail
    - Trying to use bif:contains()              Fail
    - Trying to use contains()                  Fail

The graph called geosparql.ttl, containing test data from sparql requests have been added to
the data hosted by the virtuoso graph. (added in a directory that host ttl files loaded when
the spec_to_be_loaded.sh script is launched + the launch.sh file needs to be loaded as well
for the data to be updated)

Queries tried: 

- GET the washingtonMonument point:
SPARQL prefix sf : <http://www.opengis.net/ont/sf#> SELECT ?s WHERE { ?s rdf:type sf:Point};

-- RESULT : 1 row : <url of the washington monument point>

- CHECK if the whashington monument is in the national monument polygon
SPARQL prefix sf: <http://www.opengis.net/ont/sf#> prefix geo: <http://www.opengis.net/ont/geosparql#> SELECT ?s WHERE { ?s rdf:type sf:Point ; geo:asWKT ?geoMonument . ?s2 rdf:type sf:Polygon ; geo:asWKT ?geoNationalPoly . FILTER(bif:st_intersects(?geoMonument, ?geoNationalPoly, 0)) };

This query allows to get filter only the points present in the national poly (meaning only
one point in this example). 

The Query works perfectly when writting the query above in the isql program of virtuoso in 
the virtuoso instance present in the graph VM. 

The query has also been used tested in the sparql endpoint and it WORKED ! (st_intersects 
function was recognized and the query had the expected behaviour).
